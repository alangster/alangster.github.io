Declare Variables, Not War


While every Ruby object has its purpose, variables are one of the more useful sorts, and they are rather basic in terms of their implementation. There are four main sorts of Ruby variables, and each has its own scope, or visibility, that determines how and by what the variable can be accessed. We’ll explore variables and, their scopes, in this entry.



At the top of our list, and with the broadest scope, are global variables. These variables are recognizable by their names, which begin with the $ symbol. Fittingly, global variables have what is referred to as “global scope,” meaning they are recognizable everywhere. Global variables can push right through the boundaries imposed by classes, modules, and methods, and can be accessed inside any of those. If you wish to have a variable that is used by all methods and within all classes in your program, then a global variable is a viable option. Given this fact, one would imagine that global variables are used frequently, but that is not the case. In fact, global variables are used quite infrequently in Ruby because they are not very flexible. They are given a value, and then they stay that value until changed. Part of the beauty of Ruby lies in having many different objects that work together, and global variables eliminate that together-ness.



Instance variables are another type of variable with a unique syntax (as seen above), although their scope is far more limited than that of global variables. Instance variables belong to an instance of a class, and so they are accessible by objects that are instances of that class. For each instance of the class, an instance variable may take on a different value. A common use for instance variables is in the initialize methods of classes, an example of which can be seen above. When a new object is created in the class C, that specific object is created with the instance variable set to a certain, unique value. A different object in class C can have a completely different value for the instance variable. As for their scope, only instance methods can call the instance variables.



Between global and instance variables in terms of scope are class variables. Their syntax is only slightly different from instance variables (as seen above) but they have very different uses. Class variables are shared among instances of a class, so they have a single value. In the example above, every object that is created as an instance of class C will share the class variable and its value.



Finally, there are local variables. These variables are the most limited in scope, and they lack any fancy syntax. Any Ruby program will have multiple local scopes through which it moves as the program runs. For example, every class, module, and method has its own local scope, as does the top level (i.e., beyond all definition blocks). Because they are exclusive to their local scope, one could re-use the same name for local variables across multiple local scopes and assign each variable a unique value. While this would go against best practice, it would still work, because each local variable would only be accessible within its local scope. Now, armed with this knowledge, go make some variables!
